# TacticalTablet: правила работы

## Scope

- Перед изменением кода изучать вызывающий и вызываемый код, а не только указанный файл.
- Не менять игровое поведение, если изменение поведения явно не требуется задачей.
- Не выполнять несвязанный рефакторинг.
- Не переименовывать публичные API без миграционного слоя.
- Не добавлять production-зависимости без доказанной необходимости.

## Minecraft thread safety

- Объекты `MinecraftServer`, `ServerPlayer`, `Level`, `Entity`, `ItemStack` и другие runtime-объекты Minecraft использовать только на серверном потоке, если API явно не гарантирует обратное.
- Фоновые потоки могут работать только с immutable snapshot DTO, byte arrays, strings, UUID и файловыми данными.
- Любой callback фоновой операции, изменяющий игровой мир или состояние игрока, возвращать на серверный поток.

## Persistence

- Не игнорировать ошибки чтения и записи.
- Методы сохранения должны сообщать вызывающему коду об успехе или ошибке.
- Запись выполнять через временный файл и replace/move.
- Миграции должны быть идемпотентными.
- Никогда не перезаписывать существующие пользовательские данные во время миграции без резервной копии.
- Синхронизацию с клиентом и сообщение об успехе выполнять только после успешного commit.

## Networking

- Клиентские данные считать недоверенными.
- Все C2S-действия должны проходить серверную авторизацию и проверку состояния.
- Ограничивать размеры строк, списков и других variable-length полей.
- Сохранять обратную совместимость packet ID, если версия протокола не изменяется.

## Testing

- Для исправленного дефекта добавлять regression test, когда это технически возможно.
- Чистую бизнес-логику отделять от Forge API, чтобы её можно было тестировать обычными unit tests.
- После изменений запускать минимально релевантные тесты и затем `./gradlew clean build`.
- Также запускать `git diff --check`.
- Не утверждать, что команда прошла, если она не запускалась или завершилась ошибкой.

## Completion report

В конце каждой задачи сообщать:

1. краткое описание реализации;
2. список изменённых файлов;
3. важные архитектурные решения;
4. выполненные команды и их фактический результат;
5. оставшиеся риски или непроверенные сценарии.
